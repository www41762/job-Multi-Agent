## 一、架构设计篇（Q1-Q5）

### Q1：为什么选 LangGraph 而不用纯 LangChain 的 Chain 或者 AutoGen？

**🎯 考察意图**：技术选型能力，是否真正理解各框架优劣

**✅ 参考答案**：
选 LangGraph 核心原因是**流程可控性**和**状态管理能力**：

1. **对比纯 LangChain Chain**：LangChain 的 LCEL Chain 主要支持线性 DAG，处理不了循环。但 Agent 系统必须支持循环——比如我的 JSON Schema 校验失败后，需要把报错信息喂回大模型让它自我修正，这就构成了一个环（Cycle）。LangGraph 原生支持基于图的循环和条件边，完美契合这个需求。

2. **对比 AutoGen**：AutoGen 更适合"自由对话博弈型"的多智能体场景，流程黑盒不可控。而我的系统是一个**严格的业务管道**（JD解析 → 简历分析 → 匹配打分 → 简历优化 → 面试题生成），需要精确控制每一步的执行顺序、并行逻辑和异常处理。LangGraph 的 Node/Edge/Conditional Edge 让我能精细编排这些流程。

3. **状态共享**：LangGraph 的 `StateGraph` + `TypedDict` 让所有节点共享一个全局 `AgentState`，数据流转清晰、类型安全，不需要手动在 Agent 之间传参。

---

### Q2：你的 AgentState 具体定义了哪些字段？为什么这样设计？

**🎯 考察意图**：是否真正理解 LangGraph 状态机的核心机制

**✅ 参考答案**：
我用 `TypedDict` 定义了 `AgentState`，包含以下字段：

```python
class AgentState(TypedDict, total=False):
    jd_text: str            # 原始输入：JD文本
    resume_content: str     # 原始输入：简历文本
    jd_result: dict         # JD解析Agent的结构化输出
    resume_result: dict     # 简历解析Agent的结构化输出
    matcher_result: dict    # 匹配Agent的打分与分析
    optimized_resume: dict  # 简历优化Agent的输出
    interview_qa: dict      # 面试题生成Agent的输出
    final_report: dict      # 最终汇总报告
    error: str              # 错误信息（用于异常传播）
```

设计要点：
- **`total=False`**：所有字段都是可选的。因为 LangGraph 的并行节点在不同时刻完成，不能要求所有字段同时存在。比如 `parse_jd` 节点只返回 `{"jd_result": ...}`，此时 `resume_result` 还不存在。
- **`error` 字段**：充当全局熔断开关。任何节点出错都会写入 `error`，下游节点检查到 `state.get("error")` 后直接跳过，避免雪崩式报错。
- **覆盖型更新**：我当前的字段全部是覆盖更新（每个节点返回的 dict 会 merge 到全局 State 中），没有使用 `Annotated[list, operator.add]` 这种追加型 Reducer。因为每个 Agent 的输出是独立的结构化 JSON，不需要累加。

### Q3：你的 DAG 图具体是什么拓扑结构？有几层并行？

**🎯 考察意图**：考察对工作流拓扑设计的理解深度

**✅ 参考答案**：
我的 DAG 是一个**菱形双并行拓扑**，共 4 层：

```
         START
        /      \
  parse_jd   analyze_resume    ← 第1层并行（Fan-out）
        \      /
         match                  ← 第2层汇聚（Join）
        /      \
 write_resume  generate_qa     ← 第3层并行（Fan-out）
        \      /
       summarize                ← 第4层汇聚（Join）
          |
         END
```

- **第1层并行**：JD 解析和简历解析互相无依赖，通过 `workflow.add_edge(START, "parse_jd")` 和 `workflow.add_edge(START, "analyze_resume")` 同时触发。
- **第2层汇聚**：match 节点同时接收两条边。但 LangGraph 的机制是：**每条入边到达时都会触发一次 match 节点**。所以我在 `_match_node` 里做了防御性判断——如果 `jd_result` 或 `resume_result` 还没到齐，直接 `return {}`。只有两者都到齐了才执行匹配逻辑。
- **第3层并行**：简历优化和面试题生成也无依赖，再次 Fan-out。
- **第4层汇聚**：summarize 同样做了防御性判断，等两个前置节点都完成才汇总。

### Q4：你说并行执行降低了 40% 耗时，这个数据怎么测的？基准是什么？

**🎯 考察意图**：数据是否可信，有无工程化的性能测试方法

**✅ 参考答案**：
这个数据是基于**串行 vs 并行的对照实验**得出的：

**基准（串行模式）**：
我初版是顺序执行的：JD解析（约6s）→ 简历解析（约7s）→ 匹配（约4s）→ 简历优化（约8s）→ 面试题（约8s），串行总耗时约 **33秒**。

**优化后（并行模式）**：
- 第1层并行：JD解析(6s) 和 简历解析(7s) 同时执行，取最慢的 = **7秒**
- 第2层匹配：**4秒**
- 第3层并行：简历优化(8s) 和 面试题(8s) 同时执行，取最慢的 = **8秒**
- 总计约 **19秒**

**(33-19)/33 ≈ 42%**，所以说降低约 40%。

**测试方法**：
- 在 `run_stream` 方法中，我用 `asyncio.gather(par_jd(), par_resume())` 实现并行。每个子函数内部调用了大模型的 `ainvoke` 异步接口。
- 我用 Python 的 `time.perf_counter()` 在每个步骤前后打点计时，跑了约 30-50 组不同长度的 JD 和简历，取平均值。
- 因为大模型 API 存在网络抖动，实际提升在 35%-45% 之间波动，取保守值写了"约 40%"。

### Q5：LangGraph 并行汇聚时，match 节点会被触发两次，你是怎么处理的？

**🎯 考察意图**：是否踩过 LangGraph 的并行汇聚坑，工程细节的掌握

**✅ 参考答案**：
这是 LangGraph 的一个关键机制：当多条入边汇聚到同一个节点时，**每条边到达都会触发一次该节点**。也就是说 `parse_jd` 完成后会触发一次 `match`，`analyze_resume` 完成后又触发一次 `match`。

我的处理方式是**在节点内部做防御性判断**：

```python
async def _match_node(self, state: AgentState) -> dict:
    # 1. 前置节点还没到齐，直接跳过
    if "jd_result" not in state or "resume_result" not in state:
        return {}
    # 2. 避免重复执行（已经算过就不再算）
    if "matcher_result" in state:
        return {}
    # 3. 真正执行匹配逻辑
    result = await self.matcher.match(state["jd_result"], state["resume_result"])
    return {"matcher_result": result}
```

**核心技巧**：返回空字典 `{}` 表示"本次触发不更新任何 State"，LangGraph 会自动忽略。只有当两个前置结果都到齐时，才执行真正的匹配逻辑。`summarize` 节点也用了同样的防御策略。

## 二、大模型工程化篇（Q6-Q10）

### Q6：你的 JSON Schema 校验具体怎么实现的？Schema 是手写的还是自动生成的？

**🎯 考察意图**：结构化输出工程化的落地深度

**✅ 参考答案**：
Schema 是我**手写定义**的 Python 字典，每个 Agent 都有独立的 Schema。以 JD 解析为例：

```python
JD_PARSER_SCHEMA = {
    "type": "object",
    "properties": {
        "job_parser_result": {
            "type": "object",
            "properties": {
                "job_title": {"type": "string"},
                "required_skills": {"type": "array", "items": {"type": "string"}},
                "work_years": {"type": "string"},
                "education": {"type": "string"},
                "core_responsibilities": {"type": "array", "items": {"type": "string"}},
                "bonus_items": {"type": "array", "items": {"type": "string"}},
                "industry": {"type": "string"},
                "company_type": {"type": "string"}
            },
            "required": ["job_title", "required_skills", "work_years", ...]
        }
    },
    "required": ["job_parser_result"]
}
```

校验流程：
1. 大模型返回原始文本 → `_clean_json_response()` 清理 Markdown 标记（```json 等）
2. `json.loads()` 解析为 Python dict
3. `jsonschema.validate(instance=result, schema=JD_PARSER_SCHEMA)` 强校验
4. 校验失败则进入重试循环

**如果重做这个项目，我会改用 Pydantic BaseModel 来定义**，然后通过 `model_json_schema()` 自动生成 Schema，代码更简洁且能获得 IDE 的类型提示。

### Q7：校验失败后你的重试机制是怎么设计的？是盲目重试还是有反馈？

**🎯 考察意图**：考察 LLM 输出容错的工程深度（Reflection / Self-Correction）

**✅ 参考答案**：
**绝不是盲目重试**，而是带反馈信息的"反思式重试"（Reflection-based Retry）。

核心代码逻辑：
```python
for attempt in range(MAX_RETRY + 1):
    try:
        response = await self.llm.ainvoke(messages)
        cleaned = self._clean_json_response(response.content)
        result = json.loads(cleaned)
        validate(instance=result, schema=JD_PARSER_SCHEMA)
        return result
    except (json.JSONDecodeError, ValidationError) as e:
        if attempt < MAX_RETRY:
            # 关键：把报错信息追加到对话历史中
            messages.append(HumanMessage(
                content=f"你输出的JSON格式错误，请重新输出。错误：{str(e)}"
            ))
        else:
            raise Exception(f"JD解析失败（重试{MAX_RETRY}次后）：{str(e)}")
```

**设计要点**：
1. **捕获具体错误**：区分 `json.JSONDecodeError`（JSON 语法错误）和 `ValidationError`（字段缺失/类型错误），把真实报错信息喂给模型
2. **对话历史追加**：把错误信息作为新的 `HumanMessage` 追加到 `messages` 列表中。模型在重试时能看到自己上次的错误输出 + 错误原因，利用 LLM 的自我修正（Self-Correction）能力来修复
3. **MAX_RETRY = 2**：最多重试 2 次（共 3 次尝试）。实测下来，第一次重试的修复成功率就超过 95%
4. **最终熔断**：超过最大重试次数后抛出异常，由上层 `PlannerAgent` 的 try-except 捕获，写入 `error` 字段并通知前端

### Q8：你的 `_clean_json_response` 方法只处理了 Markdown 标记，还有哪些边界情况需要考虑？

**🎯 考察意图**：考察对 LLM 输出不确定性的实战经验

**✅ 参考答案**：
当前实现只处理了 ` ```json ` 和 ` ``` ` 标记，但实际生产中 LLM 还可能有以下边界情况：

1. **前后有多余文字**：模型可能输出 "好的，以下是解析结果：{...}" 或 "```json\n{...}\n```\n希望对你有帮助"。需要用正则提取 JSON 块：`re.search(r'\{.*\}', text, re.DOTALL)`
2. **嵌套代码块**：输出 ` ```json\n```json\n{...}\n```\n``` `，需要递归处理
3. **中文标点**：字段值中出现全角引号 `""` 而非半角 `""`，导致 JSON 解析失败
4. **尾部逗号**：`{"a": 1, "b": 2,}` 这种 trailing comma，标准 JSON 不支持
5. **注释**：模型偶尔在 JSON 中加 `// 注释`

**改进方案**：可以引入更健壮的 JSON 修复库（如 `json-repair`），或者对于支持 `response_format={"type": "json_object"}` 的模型（如 GPT-4o），直接在 API 层强制 JSON 模式。

### Q9：你的 Prompt 用了哪些工程策略？为什么温度设为 0.3？

**🎯 考察意图**：Prompt Engineering 的深度和参数调优经验

**✅ 参考答案**：
我的 Prompt 策略包含以下几个层次：

1. **角色设定（Role-Play）**：每个 Agent 的 System Prompt 都以"你是专业的XX智能体"开头，给模型明确的角色定位，减少无关输出。

2. **输出模板约束**：直接在 Prompt 中给出完整的 JSON 模板，并强调"只输出JSON，不要输出其他任何文字"。这种 Few-Shot 式的模板约束比纯文字描述有效得多。

3. **STAR 法则引导**：在简历解析 Prompt 中，明确要求"按STAR法则结构化抽取项目经历（情境Situation、任务Task、行动Action、结果Result）"，引导模型输出高质量的项目描述。

4. **长期画像注入**：面试题 Agent 会动态拼接用户历史弱项信息：
   ```
   该求职者存在以下薄弱环节：{weak_str}。
   请务必包含针对这些弱项的追问题！
   ```

**温度 0.3** 的原因：
- 我的场景是**结构化信息提取**，需要确定性高、格式稳定的输出
- 温度越低，模型输出越确定，JSON 格式出错的概率越小
- 如果设为 0，可能导致输出过于刻板；0.3 是一个兼顾稳定性和灵活性的平衡点
- 对于面试题生成这种需要创造性的任务，如果效果不够丰富，可以单独调高温度

### Q10：你封装了统一的 LLM Client，如果要切换模型（比如从千问换成 GPT-4o），需要改哪些代码？

**🎯 考察意图**：抽象封装能力，多模型适配方案

**✅ 参考答案**：
我的 `llm_client.py` 封装得比较干净，切换模型**只需要改环境变量，不需要改任何代码**：

```python
def get_llm_client(streaming: bool = False):
    return ChatOpenAI(
        model=LLM_MODEL,          # 环境变量：qwen-plus / gpt-4o
        api_key=LLM_API_KEY,      # 环境变量：对应模型的 API Key
        base_url=LLM_BASE_URL,    # 环境变量：对应模型的 Base URL
        temperature=LLM_TEMPERATURE,
        streaming=streaming,
        max_tokens=4096,
    )
```

这得益于目前主流国产模型（通义千问、智谱、DeepSeek）都兼容了 **OpenAI API 格式**，所以 `ChatOpenAI` 类通过替换 `base_url` 就能直接对接。

**但如果遇到不兼容的情况，需要额外处理**：
1. 有些模型不支持 `response_format` 参数，传了会报错 → 需要在配置中增加特征开关
2. 不同模型对 `max_tokens` 的上限不同（千问 8192、GPT-4o 128k）→ 需要按模型做参数映射
3. 部分模型的 Function Calling / Tool Use 格式有差异 → 如果要用到工具调用，需要做一层适配

## 三、记忆系统篇（Q11-Q13）

### Q11：你的长短期记忆是怎么分离的？为什么不直接全部存 Redis？

**🎯 考察意图**：Memory 架构设计思路

**✅ 参考答案**：
我设计了**三层记忆架构**：

| 层级 | 存储位置 | 生命周期 | 用途 |
|------|---------|---------|------|
| 短期记忆 | AgentState（内存） | 单次分析流程 | 节点间数据流转 |
| 会话记忆 | SQLite sessions 表 | 单个会话 | 聊天历史、分析结果缓存 |
| 长期记忆 | SQLite user_profile 表 | 跨会话永久 | 用户画像（弱项、优势技能） |

**为什么用 SQLite 而不是 Redis**：
1. **部署简单**：SQLite 是嵌入式数据库，零配置，不需要额外启动 Redis 服务，降低了 Docker 部署复杂度
2. **持久化天然**：SQLite 数据直接写磁盘文件，天然持久化；Redis 需要额外配置 RDB/AOF 持久化策略
3. **足够用**：这是单用户/少用户的求职辅助工具，不存在高并发场景，SQLite 完全够用
4. **如果要上生产**：高并发场景下会替换为 Redis（热数据）+ PostgreSQL（冷数据）的组合

### Q12：你的用户画像（弱项/优势）是怎么自动提取的？什么时候写入？

**🎯 考察意图**：长期记忆的自动化闭环设计

**✅ 参考答案**：
画像提取是**自动化的**，在匹配分析完成后由 `PlannerAgent` 的 `run_stream` 方法触发：

```python
# planner_agent.py 中匹配完成后
matcher_result = await self.matcher.match(jd_result, resume_result)
if memory:
    mr = matcher_result.get("matcher_result", {})
    # 自动把匹配Agent发现的不足写入长期记忆
    for weakness in mr.get("shortcomings", []):
        memory.update_user_weakness(weakness)
    # 自动把匹配Agent发现的优势写入长期记忆
    for strength in mr.get("advantages", []):
        memory.update_user_strength(strength)
```

**闭环使用**：在后续面试题生成时，`InterviewQAAgent` 会读取这些画像数据：
```python
if user_profile and user_profile.get("weaknesses"):
    weak_str = ", ".join(user_profile["weaknesses"])
    prompt_addition = f"该求职者存在以下薄弱环节：{weak_str}。请务必包含针对这些弱项的追问题！"
```

这样用户每分析一次新的 JD，系统就会**累积**用户的弱项，后续生成的面试题会越来越有针对性。

### Q13：SQLite 的 `check_same_thread=False` 你为什么要加？不加会怎样？

**🎯 考察意图**：Python SQLite 多线程/异步并发的踩坑经验

**✅ 参考答案**：
SQLite 的 Python 绑定（`sqlite3` 模块）默认**只允许创建连接的线程使用该连接**。如果不加 `check_same_thread=False`，在 FastAPI 的异步环境中会报错：
```
ProgrammingError: SQLite objects created in a thread can only be used in that same thread.
```

原因是 FastAPI 基于 `asyncio` 和 `uvicorn`，请求可能在不同的线程池线程中执行。加上这个参数后，SQLite 允许跨线程访问同一个连接。

**但这有隐患**：SQLite 不是线程安全的，多个协程同时写入可能导致 `database is locked` 错误。我目前的解决方案是：
- 这个项目主要是单用户场景，并发写入概率极低
- 如果要支持多用户，应该换用**连接池模式**（每次操作创建新连接），或者改用 `aiosqlite` 异步库，或者直接换 PostgreSQL

## 四、流式传输与前端篇（Q14-Q16）

### Q14：你用的 SSE（Server-Sent Events）而不是 WebSocket，为什么？

**🎯 考察意图**：实时通信方案选型能力

**✅ 参考答案**：
SSE 和 WebSocket 都能实现服务端主动推送，但在我的场景中 SSE 更合适：

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 单向（服务端→客户端） | 双向 |
| 协议 | HTTP | 独立的 ws:// 协议 |
| 自动重连 | 浏览器原生支持 | 需要手动实现 |
| 适用场景 | 流式推送、实时日志 | 聊天室、协同编辑 |

我的场景是**大模型逐 Token 推送给前端**，这是典型的单向流式传输。用 SSE 的好处是：
1. **基于 HTTP**：不需要额外处理 WebSocket 的握手和心跳，Nginx 反向代理也更简单
2. **FastAPI 原生支持**：直接返回 `StreamingResponse(event_generator(), media_type="text/event-stream")` 即可
3. **前端简单**：用 `fetch` + `ReadableStream` 的 reader 就能逐行解析

如果后续要做**多人协作面试模拟**（需要双向实时通信），就该切换到 WebSocket。

### Q15：前端是怎么解析 SSE 数据流的？遇到过粘包/拆包问题吗？

**🎯 考察意图**：前端流式处理的工程细节

**✅ 参考答案**：
前端用 `fetch` + `ReadableStream` 手动解析 SSE 数据：

```javascript
const reader = response.body.getReader()
const decoder = new TextDecoder()
let buffer = ''

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  buffer += decoder.decode(value, { stream: true })
  const lines = buffer.split('\n')
  buffer = lines.pop() || ''  // 保留未完整的最后一行
  for (const line of lines) {
    if (line.startsWith('data: ')) {
      const data = JSON.parse(line.slice(6))
      onChunk(data.data)
    }
  }
}
```

**核心点**是 `buffer` 机制：
- 网络传输中，一次 `reader.read()` 拿到的数据可能是半行（拆包），也可能是多行（粘包）
- 我用 `buffer` 累积数据，按 `\n` 分割后，**最后一个元素可能是不完整的行**，用 `lines.pop()` 保留到下次循环继续拼接
- 这样就完美解决了粘包/拆包问题

另外我没用浏览器原生的 `EventSource` API，因为 `EventSource` 只支持 GET 请求，而我的 `/api/analyze` 接口需要 POST 上传 FormData。

### Q16：后端 SSE 的数据协议是你自己设计的吗？怎么区分不同类型的消息？

**🎯 考察意图**：协议设计能力

**✅ 参考答案**：
是的，我自定义了一个简单的消息协议，每条 SSE 数据是一个 JSON 对象，通过 `type` 字段区分：

```python
# 正常内容块
yield f"data: {json.dumps({'type': 'content', 'data': chunk})}\n\n"
# 流程完成信号
yield f"data: {json.dumps({'type': 'done', 'data': ''})}\n\n"
# 错误信号
yield f"data: {json.dumps({'type': 'error', 'data': str(e)})}\n\n"
```

前端根据 `type` 做不同处理：
- `content`：追加到渲染区域，实现打字机效果
- `done`：标记分析完成，解锁交互按钮
- `error`：弹出错误提示

**注意 SSE 格式要求**：每条消息以 `data: ` 开头，以 `\n\n`（两个换行）结尾，这是 SSE 协议的规范。

## 五、异常处理与健壮性篇（Q17-Q18）

### Q17：如果大模型 API 突然挂了（比如 429 限流或 500 服务错误），你的系统会怎样？

**🎯 考察意图**：系统容错和高可用意识

**✅ 参考答案**：
目前我的异常处理分为两层：

**第一层：Agent 级重试**
每个 Agent 内部有 `MAX_RETRY = 2` 的重试机制，但这主要针对**格式错误**的重试。对于网络异常（429/500），当前是直接抛出异常。

**第二层：PlannerAgent 级异常兜底**
```python
async def _parse_jd_node(self, state: AgentState) -> dict:
    try:
        result = await self.job_parser.parse_jd(state["jd_text"])
        return {"jd_result": result}
    except Exception as e:
        return {"error": f"JD解析失败: {str(e)}"}
```
异常会被捕获并写入 `error` 字段，下游节点检测到后直接跳过，最终通过 SSE 返回给前端一个 `error` 类型的消息。

**可改进的地方**：
1. 在 `llm_client.py` 中加入 **Tenacity 指数退避重试**：针对 HTTP 429/500/503 错误，等待 1s → 2s → 4s 后重试
2. 加入 **Circuit Breaker（熔断器）**：如果连续 N 次调用都失败，直接熔断，返回降级结果
3. 加入 **备用模型**：主模型不可用时，自动切换到备用模型（如千问挂了切 DeepSeek）

### Q18：如果用户上传了一个加密的 PDF 或者图片格式的简历，你的系统怎么处理？

**🎯 考察意图**：边界情况处理意识

**✅ 参考答案**：
当前的 `pdf_parser.py` 使用 PyPDF2 提取文本。对于异常输入：

1. **加密 PDF**：PyPDF2 的 `PdfReader` 遇到加密 PDF 会抛出异常，被外层 try-except 捕获，返回 HTTP 400 错误："简历文件解析失败"
2. **扫描件/图片 PDF**：PyPDF2 提取的文本会是空字符串（因为内容是图片而非文字层）。这种情况下简历分析 Agent 会收到空文本，可能产出无意义的结果

**改进方案**：
- 对 PDF 提取结果做**长度校验**：如果提取文本过短（如 < 50 字），提示用户"该 PDF 可能是扫描件，请粘贴文本版简历"
- 集成 OCR（如 Tesseract 或大模型视觉能力）处理扫描件
- 支持 `.docx` 格式（使用 `python-docx` 库）

## 六、扩展与深度追问篇（Q19-Q20）

### Q19：如果让你给这个系统加入 RAG 功能，你会怎么设计？

**🎯 考察意图**：RAG 架构认知和扩展能力

**✅ 参考答案**：
我的项目中已经有 `rag_tool.py` 的代码框架（SimpleMemoryStore），但目前还没有完全用上向量检索。如果要完善 RAG，我会这样设计：

**应用场景**：面试题生成时，从知识库中检索相关的技术知识点，作为上下文注入 Prompt。

**架构设计**：
1. **知识库构建**：收集常见面试题、技术文档、STAR 范例等，使用 Embedding 模型（如 text-embedding-3-small）生成向量
2. **向量存储**：使用 FAISS 或 ChromaDB 存储向量，支持按相似度检索
3. **检索流程**：面试题 Agent 执行前，先根据 JD 的关键技能词检索 Top-K 相关文档
4. **Prompt 注入**：将检索到的文档拼接到 System Prompt 中作为参考资料
5. **Chunk 策略**：文档按 512 Token 分块，设置 50 Token 重叠，避免语义断裂

**关键挑战**：
- 检索结果的**相关性过滤**：设置相似度阈值，避免注入无关内容增加干扰
- **Token 预算管理**：检索内容 + 原有 Prompt + JD + 简历 不能超过模型的上下文窗口

### Q20：如果要把这个系统做成多用户 SaaS 产品，你需要做哪些改造？

**🎯 考察意图**：从 Demo 到生产级系统的工程视野

**✅ 参考答案**：
目前系统是单用户 Demo 级，要做成 SaaS 产品需要以下改造：

**1. 用户系统与鉴权**
- 加入注册/登录（JWT Token 鉴权）
- 会话和画像绑定到具体用户 ID，而非全局 `default_user`

**2. 数据库升级**
- SQLite → PostgreSQL（支持并发读写）
- 聊天历史和分析结果做**分表或分库**，按用户 ID 分片

**3. 异步任务队列**
- 引入 Celery + Redis 作为任务队列
- 大模型调用是 I/O 密集型，高并发时需要任务队列削峰

**4. LLM 调用管控**
- 每个用户设置 Token 配额和 Rate Limit
- 加入 LLM 调用的计费和监控（LangSmith / LangFuse）

**5. 安全性**
- 用户上传的简历是敏感数据，需要加密存储
- 传给大模型的数据需要做 PII（个人身份信息）脱敏
- CORS 从 `allow_origins=["*"]` 改为白名单

**6. 可观测性**
- 加入链路追踪（每次 Agent 调用的耗时、Token 消耗、成功率）
- 加入告警（Agent 失败率超过阈值自动告警）
- 加入 A/B 测试（不同 Prompt 版本的效果对比）
